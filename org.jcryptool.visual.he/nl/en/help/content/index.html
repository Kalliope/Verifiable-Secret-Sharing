Homomorphic Encryption

This help file consists of three parts, namely a term clarification, scheme explanation and plugin handling instructions.

Term Clarification

A homomorphic encryption scheme is an encryption scheme which allows homomorphic operations to be performed on the ciphertext, without needing or being able to decrypt. These homomorphic operations are addition and multiplication, and performing these operations implies the same algebraic operations on the plaintext. There are three types of homomorphic encryption schemes, namely partially, somewhat, and fully homomorphic encryption schemes. The differences between these schemes are explained below.
This visualization incorporates three different schemes, namely a fully homomorphic encryption scheme by Gentry and Halevi and two partially homomorphic encryption schemes- RSA and Paillier.

Partially Homomorphic Encryption

When a scheme is partially homomorphic, either multiplication or addition can be performed on the ciphertext. In general, these schemes allow an indefinite amount of operations to be performed.

Somewhat Homomorphic Encryption

A somewhat homomorphic encryption scheme allows both homomorphic operations to be performed on the ciphertext, but the amount of consecutive operations is limited. One may interpret this as an error being introduced with every operation that is performed. When this error is small, the scheme can still correctly decrypt the ciphertext. But if many operations are performed, this error grows beyond the error-correcting ability of the scheme, which leads to incorrect decryption.

Fully Homomorphic Encryption

Fully homomorphic encryption schemes allow an indefinite amount of both homomorphic operations to be performed. This is closely related to the somewhat homomorphic encryption schemes. In addition to the normal operations, a fully homomorphic scheme allows the ciphertext to be "refreshed" without decrypting, i.e. the error in the ciphertext is reduced thus allowing additional operations whilst remaining within the error-correcting ability.

Scheme Explanation

RSA

RSA is a widely used public-key cryptosystem, which allows multiplication on the ciphertexts. It roughly works as follows (for more information see http://en.wikipedia.org/wiki/RSA). One selects two large distinct primes, p and q. The larger these primes the higher the security. Then one computes the product N of these primes. Next one chooses a public exponent e coprime to (p-1)(q-1). The coprimality is to ensure that the multiplicative inverse of e modulo (p-1)(q-1) exists. This inverse is computed using the euclidean algorithm and is called d, this is the private exponent. With this done the scheme is ready to be used, the public key consists of N and e, and the private key consists of d.
Encryption and decryption of numbers in RSA is very straightforward. To encrypt a number m, one simply computes c = m^e mod N. To decrypt a ciphertext c, one only needs to compute m' = c^d mod N = m^(e*d) mod N = m.
Now to see that RSA is partially homomorphic, consider two numbers m1 and m2 and their respective ciphertexts c1 and c2. To compute the product m1*m2 homomorphically, one simply computes the product of the ciphertexts, c1*c2. This then decrypts to m1*m2; (c1*c2)^d mod N = (c1^d)*(c2^d) mod N = (m1^(e*d))*(m2^(e*d)) mod N = m1*m2. It is also clear that this can be repeated indefinitely. The only constraint on the multiplications is that they are inherently modulo N, as is also clear from above. But since the size of N for secure encryption, this is no practical boundary.

Paillier

Paillier is an encryption scheme which allows addition on the ciphertexts. The scheme roughly is as follows (for more details see http://en.wikipedia.org/wiki/Paillier_cryptosystem. One selects two large primes p and q with the additional property that gcd(p*q,(p-1)(q-1)) = 1. This property always holds when p and q are of the same bitsize. Now one computes n = p*q and l = lcm(p-1,q-1). Next one selects a random integer g between 1 and n^2, n should divide the order of g. Then one computes mu = (L(g^l mod n^2))^(-1) mod n, where L(u) = (u-1)/n. Now the public key contains n and g, and the private key contains l and mu.
To encrypt with Paillier, one selects an integer m between 1 and n. Then one randomly selects an invertible integer r between 1 and n, and computes c = (g^m)*(r^n) mod n^2. To decrypt a ciphertext c, one simply computes m = L(c^l mod n^2)*mu mod n.
To see that Paillier is partially homomorphic, consider having two numbers m1 and m2 together with there respective ciphertexts c1 and c2. To compute the sum m1+m2 homomorphically, one needs to multiply the ciphertexts, c1*c2. This is the encryption of m1+m2: c1*c2 = (g^m1)*(r1^n)*(g^m2)*(r2^n) mod n^2 = (g^(m1+m2))*(r^n) mod n^2, and thus decrypts to m1+m2. The addition is inherently modulo n (since g^n = 1 mod n^2), but considering the size of n this is no practical boundary.

Gentry and Halevi

The fully homomorphic encryption scheme by Gentry and Halevi is a variant on the scheme by Smart and Vercauteren which in its turn is a variant on the original scheme as proposed by Gentry. To fully explain the scheme is to extensive for here, so a short version for general understanding is given. The scheme uses principal ideal lattices over a ring of polynomials modulo f(x) = x^n + 1, where n is a power of 2. As additional requirement, the Hermite Normal Form (HNF) of the lattice must be in a specific form such that it can be represented by two integers.
In the key-generation phase, one chooses a random degree n-1 polynomial v(x). This polynomial yields a rotation basis V which describes the lattice to be used. The HNF of this lattice must such that the matrix representation, i.e. the basis vectors in the columns, should equal the identity matrix except for the leftmost column. This column has as entries the determinant of the lattice and powers a root r of f(x) modulo d. Next one computes the scaled inverse of v(x), i.e. a polynomial w(x) such that v(x)*w(x) = d mod f(x). Here d equals the determinant of the lattice. In principle, one now has all the ingredients for an encryption scheme. The public key in this case is the HNF of the lattice, denoted by B. The private key consists of the rotation basis V and W. This scheme unfortunately is only somewhat homomorphic. After explaining encryption and decryption, we will explain how to fix this.
The message space for this scheme is only a single bit. To encrypt such a bit b, one generates a random n-dimensional noise vector u with entries in {-1,0,-1}. One then computes a = 2u + be1 (e1 denotes the vector (1,0,...,0)) and computes c = a mod B. But because of the special form of the HNF, this reduces c = b + 2u(r) mod d, where u(x) is the polynomial induced by the noise vector u. To decrypt a ciphertext c, one computes m = c mod V. Now again because of the special form of the HNF, this reduces to computing b = (c * wi mod d) mod 2, where wi is one term of the polynomial w(x), which must be odd. Denote this term by w.
Note that because of the special form as noted above, one can represent the public key with the integers d and r, and the private key with the integer w. The scheme is only somewhat homomorphic because it cannot refresh its ciphertexts thus reducing the noise vector. The trick to do this is called bootstrapping. Roughly the scheme is able to evaluate its own decryption algorithm homomorphically together with some homomorphic operations. This means that the scheme can evaluate some operations and then refresh the ciphertext so that it again has a low error. Roughly the encrypter "starts" decrypting. To achieve this, a "hint" on the secret integer w is given. In order to achieve this without compromising security, a large set of numbers is added to the public key of which a sparse subset sums up to this secret w. But ofcourse to be able to evaluate the decryption algorithm, the secret must be implicitly "known". Therefore the vector describing the sparse subset, i.e. a vector with a 1 in the position of the vector elements in the subset, is encrypted and also included in the public key.
Now since the secret is available homomorphically in the public key, the ciphertext can be refreshed an thus the scheme has become fully homomorphic. The operations for multiplication and addition are the same for the ciphertext. In addition, one has to refresh the ciphertext after every operation to ensure the scheme is still able to correctly decrypt the ciphertext.

Plugin Handling Instructions

To use any of the three schemes to try the homomorphic operations, one first needs a key pair. On the left side of the screen this can be done. Follow the instructions in the wizard to generate or choose a key pair. The key can also be saved for later use. For this the plugin requires a username and a password. The key is then stored in an encrypted file along with a description file. This is convenient for instance for the scheme by Gentry and Halevi if one wants to use higher security, since key generation takes a while in this case.
Once a key pair has been selected, one has to choose a modulus for the operations in the case of Gentry and Halevi. The smaller the modulus, the faster the computations will be. Then one must choose the first operand, which will be encrypted.
Now to perform homomorphic operations, two operands are necessary to perform the operation. The first operand is only required once. Then one chooses the desired operation on the left side. The second operand is requested, and the chosen operation will be performed on the two operands. This is done homomorphically, and the ciphertext result is shown. In the bottom of the screen the operations are also performed in plaintext, so the user can see what is going on.
For additional operations, the plugin uses the previous result in stead of the first operand. When one wants to verify if the ciphertext still encrypts the result, one chooses the decrypt result button on the left.
To start over with new numbers or with a new key pair choose the appropriate buttons on the left.
